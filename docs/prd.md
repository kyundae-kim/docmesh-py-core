# docmesh-py-core 외부 서비스 연결 PRD

## 1. 목적

`docmesh-py-core`는 DocMesh Python 애플리케이션이 외부 서비스를 **일관된 설정 방식**, **공통 클라이언트 팩토리**, **표준 오류/보안 정책**으로 연결하도록 만드는 공통 인프라 라이브러리다.

이 문서는 v0.1.0에서 반드시 제공해야 할 제품 요구사항을 짧고 실행 중심으로 정리한다.

## 2. 해결할 문제

현재 애플리케이션별로 외부 서비스 연결 로직을 따로 구현하면 다음 문제가 생긴다.

- 설정 방식이 서비스/앱마다 달라진다.
- 인증, 저장소, 메시징 연동 코드가 중복된다.
- 연결 실패와 재시도 정책이 제각각이다.
- 토큰, 비밀번호, DSN 같은 민감정보가 로그/예외에 노출될 수 있다.
- Keycloak 토큰 획득, JWT 검증, 사용자 정보 추출 방식이 불일치한다.

## 3. 이번 버전 목표

v0.1.0에서는 아래를 완료해야 한다.

1. 환경변수 기반 설정 로딩/검증
2. 8개 외부 서비스 클라이언트 생성 인터페이스 제공
3. 상태 확인(개별/집계) 인터페이스 제공
4. 표준 오류 모델과 민감정보 마스킹 제공
5. Keycloak 토큰 획득 및 JWT 사용자 정보 추출 제공
6. Keycloak Realm/Client/Role 프로비저닝 제공
7. 로컬/테스트/개발/운영 간 설정만으로 대상 교체 가능

## 4. 비목표

이번 버전 범위에 포함하지 않는다.

- 외부 서비스 설치/배포 자동화
- DB 스키마/마이그레이션 관리
- MinIO Bucket, Milvus Collection, NATS Stream의 업무별 생성 정책
- Ollama 모델 다운로드/수명주기 관리
- Langfuse 대시보드 구성
- 애플리케이션 도메인 로직

## 5. 대상 사용자

- DocMesh Python 백엔드 개발자
- DevOps/플랫폼 엔지니어
- 운영 담당자

## 6. 지원 서비스

| 서비스 | 역할 | 기본 확인 방식 |
| --- | --- | --- |
| Keycloak | 인증, JWT 검증, 프로비저닝 | OIDC/JWKS/Admin API |
| PostgreSQL | 메타데이터/업무 데이터 저장 | `SELECT 1` |
| SQLite | 로컬/테스트 저장소 | `SELECT 1` 또는 파일 접근 |
| MinIO | 객체 저장소 | 서버 응답 또는 Bucket 확인 |
| Milvus | 벡터 저장/검색 | 연결 상태 확인 |
| Ollama | 모델 호출 | 버전/모델 목록 조회 |
| Langfuse | 트레이싱/관측성 | 인증 API 요청 |
| NATS | 메시징 | Ping/Pong |

## 7. 핵심 요구사항

### PR-1. 설정

- 모든 연결 정보는 환경변수에서 읽어야 한다.
- 시작 시점에 설정을 검증해야 한다.
- 공백 문자열은 미설정으로 처리해야 한다.
- Boolean/숫자형 값은 타입과 범위를 검증해야 한다.
- 민감정보는 문자열 표현과 로그에서 마스킹해야 한다.

### PR-2. 클라이언트 팩토리

- 각 서비스 클라이언트는 독립적으로 생성할 수 있어야 한다.
- 지연 연결을 기본으로 지원해야 한다.
- 필요한 서비스만 선택적으로 생성할 수 있어야 한다.
- 종료가 필요한 클라이언트는 명시적 종료 인터페이스를 제공해야 한다.
- PostgreSQL과 SQLite는 같은 상위 저장소 진입점에서 선택 가능해야 한다.

### PR-3. SQLite 지원

- 파일 경로와 `:memory:`를 지원해야 한다.
- 상대경로는 작업 디렉터리 기준으로 해석해야 한다.
- 파일이 없으면 생성 가능해야 한다.
- 상위 디렉터리가 없으면 명확한 오류를 제공해야 한다.
- 읽기 전용, WAL, Busy Timeout을 환경변수로 제어할 수 있어야 한다.

### PR-4. 상태 확인

- 서비스별 상태 확인 함수를 제공해야 한다.
- 전체 서비스 집계 상태 확인을 제공해야 한다.
- 결과에는 서비스명, 성공 여부, 응답 시간, 마스킹된 오류 원인이 포함되어야 한다.
- 선택 서비스 장애가 시작 실패로 이어질지 설정할 수 있어야 한다.

### PR-5. 오류/로깅

- 설정 오류, 인증 오류, 연결 거부, DNS 오류, 타임아웃을 구분해야 한다.
- 필수 환경변수 누락 시 변수명을 포함한 오류를 반환해야 한다.
- 설정 오류와 영구 인증 오류는 자동 재시도하지 않아야 한다.
- 서비스별 재시도/백오프 정책을 적용할 수 있어야 한다.
- 연결 시도/성공/실패/재시도/종료를 구조화 로그로 남길 수 있어야 한다.
- 비밀번호, 토큰, Secret Key, 전체 DSN/URI는 로그/예외에 남기면 안 된다.

### PR-6. Keycloak 프로비저닝

- Realm, Client, Realm Role, Client Role 생성/갱신을 지원해야 한다.
- 반복 실행 시 멱등해야 한다.
- 선언에서 제거된 리소스를 자동 삭제하면 안 된다.
- Dry-run을 제공해야 한다.
- 결과를 생성/갱신/변경 없음/실패로 구분해야 한다.

### PR-7. Keycloak 인증

- OIDC Token Endpoint 기반 Access Token 획득 인터페이스를 제공해야 한다.
- 기본은 Client Credentials Grant여야 한다.
- 명시적 설정 시 Password Grant를 허용할 수 있어야 한다.
- JWT 검증 후 `sub`, `preferred_username`, `email`, `given_name`, `family_name`, `name`, `realm_roles`, `client_roles`, `claims`를 표준 필드로 반환해야 한다.
- 토큰 원문은 로그/예외에 노출되면 안 된다.

## 8. 보안 원칙

- 민감정보는 환경변수 또는 Secret 관리 기능으로 주입한다.
- 운영 환경에서는 TLS 검증을 기본 활성화한다.
- TLS 비활성화는 개발/테스트에 제한한다.
- DSN/URI 출력 시 사용자명, 비밀번호, 토큰, 민감 Query Parameter를 마스킹한다.
- Keycloak 프로비저닝은 최소 권한 Service Account 사용을 권장한다.

## 9. 비기능 요구사항

- Python 3.11 이상 지원
- 설정 파싱/설정 객체 생성은 일반 환경에서 100ms 이내 목표
- 상태 확인은 병렬 실행 가능해야 함
- 선택 서비스 장애가 다른 서비스 생성에 영향을 주지 않아야 함
- 반복 연결/종료 시 리소스 누수가 없어야 함
- 실제 연동 테스트는 통합 테스트로 분리해야 함

## 10. 완료 기준

아래를 모두 만족하면 v0.1.0 완료로 본다.

1. 8개 서비스 설정 객체와 클라이언트 생성 인터페이스 제공
2. 필수 환경변수 누락/타입 오류 테스트 통과
3. 정상 연결/대표 실패 케이스 통합 테스트 통과
4. SQLite `:memory:`, 파일 경로, 읽기 전용, 잘못된 경로 테스트 통과
5. 민감정보 비노출 테스트 통과
6. 상태 확인 및 종료 인터페이스 제공
7. 예시 환경변수 문서화 완료
8. CI 정적 검사, 단위 테스트, 민감정보 노출 검사 통과
9. Keycloak 프로비저닝 생성/멱등성/Dry-run/부분 실패 테스트 통과
10. Keycloak 토큰 획득 및 JWT 검증/추출 테스트 통과
